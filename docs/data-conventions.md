# Data conventions: dates, times, and effective-dated history

This document defines how sym names and interprets every date/time field. It is a
contract: new columns follow these rules, and the names are chosen to migrate
cleanly to other engines (Snowflake, BigQuery) without quoting.

## 1. Two time types, never mixed

| Concept | Type (Postgres) | Maps to (Snowflake) | Meaning |
| --- | --- | --- | --- |
| **Business / calendar date** | `DATE` | `DATE` | A calendar day. Trading dates are **exchange-local** (from the per-MIC `trading_calendar`). Timezone-naive — a date is a date. |
| **System / audit instant** | `TIMESTAMPTZ` | `TIMESTAMP_TZ` | A point in time, stored **UTC**. Generated by `now()` / `datetime.now(UTC)`. "When the pipeline did this," never a business date. |

So there is never a UTC-vs-local question on a single field: business time is a
naive `DATE`; system time is a UTC instant. A cross-exchange "as-of *calendar
date* D" means "each exchange's latest session ≤ D," resolved via its own
calendar.

> Nuance: the CLI's "run as-of" uses Python `date.today()` = the **machine-local**
> date. It only labels a run; it never affects stored trading data (those dates
> come from the calendar). Pin it to UTC if strict run-labeling matters.

## 2. Naming rules

- **Instant → `*_at`** (`created_at`, `updated_at`, `recorded_at`, `run_at`,
  `started_at`, `finished_at`, `detected_at`, `reviewed_at`, `resolved_at`,
  `checked_at`, `decided_at`, `retrieved_at`).
- **Calendar date → `*_date`** (`session_date`, `ex_date`, `delist_date`,
  `cursor_date`, `effective_date`, `as_of_date`, `first_seen_date`,
  `last_seen_date`, `floor_reached_date`, `first_session_date`,
  `last_session_date`).
- **Point-in-time observation date → `as_of_date`** (the one canonical name for
  "the date a fact/observation is *for*"): `fact_returns.as_of_date`,
  `fundamentals.as_of_date`, `universe_accuracy_check.as_of_date`. This name is
  canonical at **every layer**, not just columns: Python parameters, local
  variables, and CLI flags all use `as_of_date` / `--as_of_date` (never `asof`,
  `as_of`, a `today` param that accepts any date, or a bare `date`). A literal
  `date.today()` *default* may stay named `today` — the value really is today.
- **Sanctioned exceptions** (kept as the universal SCD valid-time idiom, *not*
  forced to `_date`): **`valid_from`**, **`valid_to`**, and `pit_valid_from`.
- **No column is ever a bare type or reserved word** (no `date`, `timestamp`),
  and **identifiers are never quoted** — so they case-fold cleanly Postgres
  (lowercase) → Snowflake/BigQuery (uppercase). This is the single most important
  portability rule.

## 3. Effective-dated history (SCD Type 2) and the half-open interval

Slowly-changing attributes — symbology, company names, GICS, universe membership
— are stored **SCD Type 2**: a change closes the old row and opens a new one,
each tagged with a validity span `(valid_from, valid_to)`. The immutable key
(e.g. `composite_figi`) never changes; history is preserved, never overwritten.

**Intervals are half-open: `[valid_from, valid_to)`.**
- `valid_from` is **inclusive** — the new row owns the change day.
- `valid_to` is **exclusive** — the old row is valid *up to but not including* it.
- `valid_to IS NULL` means **currently valid / open-ended**.
- One row's `valid_to` equals the next row's `valid_from` (enforced: a
  `valid_to > valid_from` CHECK + a `btree_gist EXCLUDE` over
  `daterange(valid_from, valid_to, '[)')` → no overlap, no gap).

This is why the column is `valid_to`, not `end_date`: `end_date` reads as
*inclusive* ("valid through"), which invites an off-by-one (`BETWEEN start AND
end` double-counts the boundary day). `valid_to` carries the exclusive bound.

The as-of query is always:

```sql
WHERE valid_from <= :d AND (valid_to IS NULL OR valid_to > :d)
```

### Worked example — Block's ticker change (SQ → XYZ)

Block Inc. changed its ticker from `SQ` to `XYZ` on **2025-07-23**, but it is the
same company — one immutable `composite_figi = BBG0018SLC07`. In
`security_symbology`:

| composite_figi | symbol_value | valid_from | valid_to |
| --- | --- | --- | --- |
| BBG0018SLC07 | `SQ` | 2015-11-19 | **2025-07-23** |
| BBG0018SLC07 | `XYZ` | **2025-07-23** | `NULL` (current) |

Resolving "which ticker on date D?" with the as-of query:

| D | SQ row (`valid_to=2025-07-23`) | XYZ row (`valid_from=2025-07-23`) | Result |
| --- | --- | --- | --- |
| 2020-01-01 | `2025-07-23 > 2020-01-01` ✓, in range | not yet started | **SQ** |
| **2025-07-22** | in range (last valid day of SQ) | not yet started | **SQ** |
| **2025-07-23** | `2025-07-23 > 2025-07-23` ✗ → excluded | `valid_from <= D` ✓, `valid_to NULL` → **included** | **XYZ** |
| today | excluded | included | **XYZ** |

So on the boundary day **2025-07-23 the answer is `XYZ`** — the new row owns the
change day (inclusive `valid_from`), and SQ's effective last day is 2025-07-22
(exclusive `valid_to`). The boundary day belongs to exactly one row, with no gap
and no overlap. The FIGI is unchanged throughout, so prices and returns keyed on
`composite_figi` span the rename seamlessly.
