# Story 3.1-ext: Return-window expansion + taxonomy v2

Status: done (window expansion + code-review resolution incl. the TR dividend-basis bug fix)

## Story

As a researcher, I want the standard performance horizons (short-term day counts,
cumulative multi-year totals, the last completed quarter) available in the returns
matrix, with a `kind` taxonomy that doesn't mislead, so I can pull any common window
for assets *and* benchmarks (and compute alpha on them) without bolting on ad-hoc SQL.

## Acceptance Criteria

1. **5D / 10D** — trading-session-count windows (N sessions back), cumulative.
2. **Cumulative multi-year + since-inception** — `2Y, 3Y, 5Y, 10Y, 20Y, 30Y, SI` total
   returns (cumulative siblings of the annualized `*_ANN` windows; same endpoints, ratio−1 vs CAGR).
3. **PQ** — last completed calendar quarter, a *discrete* period return with **both endpoints
   in the past** (end = prior quarter's last session, base = the quarter before it).
4. **Taxonomy v2** — `rolling`+`multiyear` unified into `trailing` (the split was only
   lookback-unit + the already-separate `annualized` flag); `ipo`→`inception` (general — an
   index/fund has an inception date, not an IPO), codes `IPO`/`IPO_ANN`→`SI`/`SI_ANN`. Final
   kinds: `calendar | session | trailing | inception | period`.
5. Windows are **appended** (ids 19–28), never renumbered, so the 9M+ existing `fact_returns`
   rows stay valid; existing windows' values + `input_hash` are byte-identical after the change.
6. Applies to both equity (`fact_returns`) and index (`fact_index_returns`) returns, so alpha
   works on every new window. DB-free tests + the live SM-6 accuracy gate cover them.

## Dev Notes

- **Why `end_date`**: the matrix was strictly base→as-of. PQ needs a past *end*, so the
  numerator session is generalized via `end_date` (default as-of). Non-period windows are
  unchanged because `end == asof`, keeping `input_hash` stable → dirty-set skips the rewrite
  of existing rows (only the new windows insert).
- **`kind` is plumbing**: read only by base-date computation, never exposed in queries
  (filter by `code`/`annualized`). The only financial split is `calendar` (to-date, resets)
  vs trailing; cumulative-vs-CAGR is the `annualized` flag.
- Since-inception windows (`SI`, `SI_ANN`) anchor on the ~1990 calendar floor, which predates
  the accuracy fixture's 31y Yahoo window, so they're the allowed-uncovered codes in the gate.
