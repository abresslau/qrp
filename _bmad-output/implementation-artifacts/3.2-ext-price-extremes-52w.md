# Story 3.2-ext: 52-week high/low price extremes (calc layer)

Status: done

## Story

As a researcher,
I want the trailing 52-week high and low (and how far the current price sits off each)
materialized daily for every security in the universe **and** every benchmark index,
computed in the same pass as returns,
so that I can screen "near 52w high / breaking down to new lows" cross-sectionally without
bolting on ad-hoc window SQL over 13.5M price rows.

## Acceptance Criteria

1. **Equity extremes table** — `fact_price_extremes` exists as a loader-written table (NOT a
   materialized view), PK `(composite_figi, as_of_date)`, columns: `high_52w`, `low_52w` (adjusted
   prices, NUMERIC), `high_52w_date`, `low_52w_date` (the session each extreme was set), `pct_off_high`,
   `pct_off_low` (NUMERIC decimals), `input_hash` (NOT NULL), `gated` (BOOLEAN), `created_at`/`updated_at`,
   FK on `composite_figi → securities`. Cross-sectional index on `(as_of_date)`.
2. **Index extremes table** — `fact_index_extremes`, PK `(sym_id, as_of_date)`, same metric columns
   (`high_52w`, `low_52w`, `high_52w_date`, `low_52w_date`, `pct_off_high`, `pct_off_low`, `input_hash`,
   timestamps), FK on `sym_id → instrument`, cross-sectional index on `(as_of_date)`. No `gated` column
   (index levels carry no `prices_review` gate). Mirrors the `fact_returns` ↔ `fact_index_returns` split.
3. **Definition** — for each (security, as_of_date), the 52-week window is the trailing **365 calendar
   days inclusive** of `as_of_date`: all sessions `d` with `as_of_date − 365d ≤ d ≤ as_of_date`.
   `high_52w`/`low_52w` = max/min **adjusted close** over those sessions; `*_date` = the **most recent**
   session that achieved that extreme (ties broken to the later date — "how long since it last printed
   the high"). `pct_off_high = adj_close[as_of] / high_52w − 1` (≤ 0), `pct_off_low = adj_close[as_of] /
   low_52w − 1` (≥ 0). Securities with < 52 weeks of history use the extreme over **available** trailing
   sessions (since-inception within the window) — no special NULL, no partial-window flag. Indexes use
   `index_levels.level` in place of `adj_close`, same definition.
4. **Computed in the returns pass (no extra price reads)** — the equity extremes are produced inside
   `load_returns`'s existing per-figi loop (which already pulls the adjusted series, sessions, and gate
   set) and the index extremes inside `recompute_index_returns`'s per-series loop (which already pulls
   levels). No new full-view scans; the sliding-window extremum is **O(sessions)** per security via a
   monotonic deque, not O(sessions × as_of_dates).
5. **Provenance + dirty-set + delete-orphan parity with `fact_returns`** — each row stamps
   `input_hash = sha256(calendar_version | as_of_date | high_52w | high_52w_date | low_52w | low_52w_date | price[as_of])`;
   the COPY→UPSERT rewrites only rows whose `input_hash` (or `gated`) changed; as_of_dates that lost their
   price (e.g. an `--overwrite` removed a phantom bar) are DELETEd, exactly as `load_returns` already does.
   (Amended 2026-06-20 by code-review Decision 1: the trailing `price[as_of]` term — parity with
   `fact_returns` hashing its endpoint prices — ensures a same-day close correction that does not move the
   trailing extreme still re-dirties the row, since `pct_off_*` depends on the current price.)
6. **Gating (AR-9, equity only)** — a row is `gated=True` (extremes held NULL) iff `as_of_date`,
   `high_52w_date`, or `low_52w_date` is an **unreviewed** `prices_review` flag date (reuse the existing
   `_unreviewed_flag_dates` set). A flag elsewhere in the window that did **not** set the extreme does not
   gate (precise, unlike a window-wide gate). `input_hash` still reflects the prices so a later price change
   re-dirties even a gated row.
7. **Wired into recompute + EOD, no new step** — `sym recompute` (equity) and `sym benchmarks` /
   `sym msci-import` (index) emit the extremes alongside returns automatically; the EOD `recompute` and
   `benchmarks` steps therefore cover them with **no new `EodStep`**. The recompute/benchmarks summary
   lines report the extreme row counts.
8. **Tests** — DB-free unit tests for the pure extremum function (sliding-window correctness incl. ties,
   partial history, single session, pct-off signs), gating precision (flag on the extreme date vs an
   inert flag elsewhere), `input_hash` determinism + sensitivity, and an anti-orphan/dirty-set assertion.
   Live verification: recompute a bounded recent range and confirm a known name's 52w high/low matches a
   hand-computed reference and is plausible vs Yahoo (report-only — see Dev Notes on the intraday caveat).

## Dev Notes

### Where this fits (the calc layer, Epic 3 / AR-7)

The materialized **returns layer** (`fact_returns`, AR-7) is a loader-written table filled by
`packages/sym/src/sym/returns/loader.py::load_returns`, which reads `v_prices_adjusted` **filtered
per-figi** (the Story 3.3 directive — a live full-view scan is ~9 min over 13.5M rows). The index
sibling is `packages/sym/src/sym/benchmarks/returns.py::recompute_index_returns` over `index_levels`.
52-week high/low is a *new metric on the same rails*: it is **not** a return over a window (so it does
NOT belong in `fact_returns`/`return_window` — don't add a window id), but it is computed from the same
per-figi adjusted series the loader already has in memory. **Compute it in the same loop** — re-reading
prices for a second pass would double the most expensive part of recompute.

### Reuse — do NOT reinvent

- **`load_returns` per-figi loop** (`loader.py:289-317`) already pulls `price_rows` → `adj`,
  `dividends`/`tri`, `sessions`, `gated_dates`, and computes `as_of_dates`. Hook the extremes in there;
  pass `adj` + `as_of_dates` + `gated_dates` to the new computation. Do not open a second cursor pass.
- **`recompute_index_returns` per-series loop** (`benchmarks/returns.py:107-118`) already pulls `levels`
  + `sessions` + `as_of_dates`. Hook index extremes there.
- **`_unreviewed_flag_dates(conn, figi)`** (`loader.py:226`) is the AR-9 gate set — reuse it verbatim.
- **`current_calendar_version`** (`sym.calendar.snapshot`) — reuse for the hash, same as returns.
- **`set_updated_at()`** trigger function already exists (used by both fact tables) — attach it; do NOT
  redefine it.
- **COPY→temp→UPSERT-with-dirty-set** pattern — copy `_upsert` in `loader.py:236-265` exactly (temp table
  `ON COMMIT DROP`, `INSERT … ON CONFLICT … DO UPDATE … WHERE input_hash IS DISTINCT FROM …`). This is
  what lets the nightly recompute skip unchanged rows.

### The one real design decision — close-based (default) vs intraday

The headline "52-week high" quoted on stock pages is the **intraday** high (the `high` OHLC column). This
story computes the extreme on the **adjusted *close*** because (a) `v_prices_adjusted` exposes only
`close_raw`/`adj_close` — there is no `adj_high`/`adj_low`, so an intraday variant would require extending
the view to derive `high/low ÷ split_factor` (added scope + a migration), and (b) the entire calc layer is
close-based (returns use `adj_close`), so a close-based extreme is consistent and split-correct for free.
**Implement close-based.** Consequence for the live check: Yahoo's `fiftyTwoWeekHigh/Low` is intraday, so
expect close-based to sit *inside* Yahoo's range — treat the Yahoo comparison as **report-only / plausibility**,
not an exact gate. An intraday variant (extend `v_prices_adjusted` with `adj_high`/`adj_low`, add
`high_52w_intraday`/`low_52w_intraday`) is a clean follow-up if Andre wants the headline number — flagged in
the open question below, NOT built here.

### Efficiency (don't make recompute slower)

Recompute already iterates ~2,035 securities × ~250 as_of_dates × 28 windows. The naive extreme
(re-scan the trailing window for every as_of) is O(sessions²) per security and would dominate. Use a
**two-deque sliding window** (one monotonic-decreasing deque for the max, one monotonic-increasing for the
min) advancing through the ascending session list once: O(sessions) total per security, with the
trailing-365d eviction at the deque front. This is the standard sliding-window-maximum algorithm; keep
it pure and unit-test it independently of the DB.

## Open Questions (for Andre — do not block implementation)

1. **Close-based vs intraday 52w high/low.** Story implements **close-based** (adjusted close) for
   consistency with the calc layer and because `v_prices_adjusted` exposes no adjusted high/low. If you
   want the *headline* intraday number (matches Yahoo/Bloomberg), it's a clean follow-up: extend
   `v_prices_adjusted` with `adj_high`/`adj_low` and add intraday extreme columns. Flag if you want that
   folded in instead.
2. **Window length convention.** Default is **trailing 365 calendar days** (≈ "1 year", matching the 1Y
   return window's spirit). If you specifically mean 52×7 = 364 days, or a 252-trading-session count, it's
   a one-line constant change.
3. **Tie-break on the extreme date.** Default records the **most recent** session that hit the extreme
   ("how long since it last printed the high"). Say if you'd prefer the *first* occurrence.
