# Decision Log — sym Product Brief

## Session: 2026-05-19

### D-001 — Permanent ID anchor: OpenFIGI
**Decision:** Use OpenFIGI (FIGI / CompositeFIGI) as the permanent identifier spine.
**Rationale:** FIGI is an open standard, free API, doesn't change with ticker or SEDOL changes. Survives vendor transitions (yfinance → EODHD → Bloomberg/FactSet). User's explicit requirement: the permanent ID concept must outlast any data vendor relationship.

### D-002 — Data sourcing strategy: yfinance for dev, paid source for production
**Decision:** yfinance as the initial data source for development and prototyping. Documented migration plan to a paid source (EODHD ~$20/month identified as leading candidate) before production.
**Rationale:** Budget constraint — cannot afford paid data now. yfinance has known issues (rate limiting at ~950 tickers/session, ToS concerns for commercial use, data quality gaps on corporate actions). These risks are accepted for dev phase with explicit plan to migrate.
**Risks accepted:** Survivorship bias gaps, silent corporate action errors, Yahoo ToS exposure during dev phase.

### D-003 — Geographic scope: US + Developed Markets + Brazil
**Decision:** Scope is US equities, developed markets (major exchanges), and Brazil (B3). Not all emerging markets.
**Rationale:** User specified US + Developed as initial scope, then added Brazil explicitly. Signals intent to include select EM markets from the start, not a pure developed-only mandate.

### D-004 — Investable universe size: 5,000–10,000 securities
**Decision:** Target investable universe, not all listed securities.
**Rationale:** User is building a quant research tool, not a market data vendor. A liquidity/size filter to ~5k–10k names is appropriate. Exact filter criteria deferred to implementation.

### D-005 — Tech stack: Python + DuckDB/Parquet, PostgreSQL later
**Decision:** Python as the primary language. DuckDB + Parquet as local-first architecture. PostgreSQL as a future option when the Quant Research Tool requires a proper server.
**Rationale:** DuckDB suits the analytical workload (batch writes, heavy reads, time-series queries). Parquet files are portable and DuckDB-queryable. User mentioned both PostgreSQL and DuckDB; recommended DuckDB-first with clear migration path.

### D-006 — History depth and update cadence
**Decision:** 10 years of daily EOD price history at launch. Daily batch update cadence.
**Rationale:** User confirmed 10Y EOD as correct assumption.

### D-007 — Classification: GICS preferred
**Decision:** Use GICS (or GICS-approximated) sector/industry classification.
**Rationale:** User explicitly requested GICS. Free source: financedatabase (GitHub) provides GICS-approximated classifications. Production-grade GICS from MSCI/S&P requires licensing.

## Session: 2026-05-20

### D-008 — Commercial latency surfaced in brief
**Decision:** Make sym's latent commercial intent explicit in the brief (Executive Summary closing line and Vision). Reclassify yfinance from a "dev-phase accepted risk with migration plan before firm production use" to a **hard precondition gate**: yfinance cannot be the active data source for any commercial activity (distribution to third parties, paid analysis derived from sym, or production use at a firm). Reframe vendor independence so the same abstraction that lets the author migrate vendors also lets a future buyer bring their own licensed feed.
**Rationale:** Author clarified that sym is a personal project today but is intended to be used by him later and possibly sold to companies. Prior brief language ("internal equity security master for a quantitative investment research practice," "before production use at a firm") was personal/firm-oriented and silent on productization. That silence would have caused downstream PRD and architecture work to optimize for personal-use tradeoffs only. Naming the commercial latency preserves design optionality — schema-as-contract, vendor-pluggable sourcing, classification choices that survive licensing constraints — without committing to ship a commercial product.
**Scope of brief edits applied:** (1) Executive Summary closing sentence on commercial latency; (2) Vision paragraph expansion on productization optionality; (3) "Vendor independence by design" — production → "any commercial use," added buyer-pluggable framing; (4) "Data sourcing — primary risk" — yfinance reclassified as hard commercial gate.
**Items intentionally NOT changed in this pass (deferred):** Q1 universe filter criteria (liquidity/market-cap thresholds, country-specific vs. universal, point-in-time vs. current). Q2 history immutability vs. vendor-swap divergence ("historical records are unaffected" claim left in §Solution despite being technically misleading once vendor adjusted-price math diverges). Q3 point-in-time GICS (currently silent). Q5 researcher-outcome + commercial-readiness success criteria (current criteria are infra-pass/fail only). Author chose patch path (option a) over walking all five questions in this session.
