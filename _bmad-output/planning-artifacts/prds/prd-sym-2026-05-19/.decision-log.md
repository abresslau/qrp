### D-001 — Database: PostgreSQL (not DuckDB)
**Decision:** PostgreSQL as the primary database engine.
**Rationale:** Total data volume ~5–10 GB (10k securities × 10Y daily prices + returns). DuckDB/Parquet earns its complexity at hundreds of GB+. PostgreSQL is simpler, DBeaver-native, REST API ready, and appropriate for this scale.

### D-002 — Returns: Both total return and price return stored separately
**Decision:** Store both price return (PR) and total return (TR, dividends reinvested) for all return windows.
**Rationale:** For L/S portfolio construction and factor research, total return is the correct standard. Price return is retained for compatibility with price-only sources and comparisons. User explicitly requested both.

### D-003 — Return methodology: FactSet EXDATE_C equivalent
**Decision:** Total return uses ex-date dividend reinvestment (FactSet EXDATE_C equivalent). Gross return (no withholding tax adjustment).
**Rationale:** FactSet research confirmed EXDATE_C is the standard for equity total return. Withholding tax netting not supported by free data sources.

### D-004 — Return windows: FactSet-equivalent definitions
**Decision:** 18 windows matching FactSet's return snapshot schema.
- Calendar-anchored: 1D, WTD, MTD, QTD, YTD
- Rolling same-calendar-date: 1M, 3M, 6M, 9M, 1Y
- Annualized compound (CAGR): 2Y, 3Y, 5Y, 10Y, 20Y, 30Y, IPO
**Rationale:** Direct equivalence to FactSet enables validation against professional data and future migration.

### D-005 — Pipeline phases: incremental approach
**Decision:** Three phases — (1) small dev load, (2) full historical backfill over multiple days, (3) daily delta updates.
**Rationale:** User wants to validate schema on small set before committing to full historical load.

### D-006 — Visualization: DBeaver / DB explorer, no custom UI
**Decision:** No custom UI in v1. Schema designed to be DBeaver-friendly.
**Rationale:** User is comfortable with database explorers. Custom UI deferred.

### D-007 — sym scope boundary
**Decision:** sym v1 = security master + market data + returns. Universe management = separate future module. sym exposes FIGI as the universal join key for other warehouse modules.
**Rationale:** User confirmed "universe treated as separate database." sym is Module 1 of a broader data warehouse.

### D-009 — Exchange scope: all major developed markets
**Decision:** Developed market exchange scope = "all major developed market exchanges." Specific exchange list defined at implementation for FR-12.
**Rationale:** User confirmed scope is not limited to a named subset.

### D-010 — Finalization: 2026-05-20

### D-013 — Added SM-6 returns-accuracy regression gate
**Decision:** New Primary success metric SM-6 (§7) plus counter-metric SM-C2. SM-6: sym PR/TR vs an independent published reference series across all 18 windows for the ~50 adversarial benchmark names, per-window tolerance (~5 bps clean, looser for corporate-action-heavy), run as a **regression gate on every returns-engine change** — not a one-time launch check. SM-C2 forbids widening tolerances to force a pass.
**Rationale:** Architecture validation found SM-1..SM-5 are all infrastructure pass/fail; none validate that returns are *correct*. A subtle TR reinvestment-timing error would pass every existing metric. Identified during architecture Step-4 party-mode and re-surfaced in Step-7 validation gap analysis. SM-6 generalizes the existing SM-3 (1Y / 10 US names / TR-only spot-check) into a full-window correctness gate.
**How to apply:** SM-6 gates every change to the returns engine (`v_prices_adjusted`, factor derivation, `fact_returns` recompute, window definitions). SM-3 retained as a narrower launch spot-check. Reference series per [A-4] requires a separately-pulled independent TR/PR source (existing FactSet sample is price-return only).
**Not reversing:** purely additive; no prior decision changed.

### D-014 — NFR-1 reconciled to two-stage anomaly semantics; NFR-2 reframed for three-layer model
**Decision:** Rewrote NFR-1 (§10) and reframed NFR-2 (§10).
- **NFR-1:** intent preserved (suspect ±50% moves must not contribute to published returns until reviewed), mechanism reconciled to the architecture's D2 two-stage rule — annotate the suspect price at ingestion (recorded, flagged, pipeline never halts, legitimate large moves confirmed not discarded), gate its returns at the point of publication. Kept mechanism-light (capability, not table names); pointer to D2 for the how.
- **NFR-2:** the original `adjusted_close > close` check assumed a stored vendor adjusted_close, which the three-layer model (raw prices + explicit factors → derived adjusted) no longer stores. Reframed as the determinism/reproducibility check on the derivation: a derived adjusted value that can't be reproduced from raw + explicit factors is the data-error flag.
**Rationale:** Architecture D2 ("annotate, don't block" + "gate before `fact_returns` recompute") is a reconciliation of NFR-1's intent, not a reversal — but the original NFR-1 wording ("not written to the returns tables until manually reviewed") implied compute-then-withhold and named no annotate-at-ingestion behavior, so it read as contradicting the architecture. NFR-2 was flagged obsolete-as-written by the same architectural shift (Step-7 validation, NFR findings).
**Scope note:** NFR-2 was adjacent to (not named in) the original change signal; reconciled in the same pass after surfacing it to the author rather than leaving a knowingly-stale NFR. Author did not object.
